# StructureRAG: Structure-Aware Retrieval + Runtime Confidence Scoring

## Overview

StructureRAG is a reference implementation demonstrating how to build production-grade RAG systems that handle structured documents correctly and provide runtime confidence signals for every answer.

This project directly addresses the core reliability gap in modern RAG pipelines: standard chunking and retrieval strategies fail on documents with inherent structure (tables, financial data, contracts with cross-references), and existing systems provide no runtime signal of when these failures have occurred.

---

## The Problem Space

### Why Standard RAG Fails on Structured Documents

#### Problem 1: Standard RAG Destroys Document Structure

When you ingest a PDF, Word doc, or enterprise document into a standard RAG pipeline, the chunking step treats everything as flat text. A table with three columns and twenty rows becomes a block of space-separated text with no relationship between cells. A chart with a label, axis, and data series becomes a caption and some numbers with no structure. A sentence that says "see Table 3 on page 12" resolves to nothing because there is no graph of cross-references — there is only text.

The consequences are compounding:

- Retrieval returns fragments. The retrieved chunk is a piece of a table, not the table.
- Answers are wrong or incomplete because the model never saw the full structured context.
- Provenance is broken. You cannot trace an answer back to the exact cell or clause that supports it.
- Reliability degrades as the document corpus grows because structural relationships become more important, not less.

Standard RAG pipelines handle this by ignoring it. They chunk on token count, embed, retrieve, and hope the context window fills in the gaps. It works in demos where the document is a simple FAQ. It fails in production where the document is a 200-page contract, a financial report, or a technical specification.

#### Problem 2: There Is No Trust Signal at Runtime

The second problem is that LLMs sound confident whether they are right or wrong. A model will state an incorrect number from a corrupted table retrieval with the same tone as a correct number from a clean retrieval. Evals pass because the eval questions were written by the same team that wrote the documents. Production breaks because real users ask edge-case questions that expose gaps.

Existing approaches to this problem evaluate the output only. They run a judge model on the final answer and ask "does this sound correct?" This misses everything that happened upstream: was the retrieval step complete? Was the right table chunk returned? Did the model hallucinate a number that looks plausible but has no source in the document?

A real confidence signal has to instrument the pipeline, not just the output.

#### Problem 3: No Audit Trail Means No Accountability

In production AI systems — especially in legal, finance, compliance, and enterprise settings — the question is not just "what did the AI say?" but "why did it say that?" You need to be able to show a compliance officer exactly which clause was retrieved, which model step produced the summary, and what the confidence score was at each stage. Standard pipelines produce none of this.

## The Solution We Are Building

A two-module Python backend with a clean minimal frontend that demonstrates how to solve structured document retrieval and runtime confidence scoring in practice.

### Module 1: Structure-Aware Document Ingest + Retrieval (The Context Pillar)

**What it does:**

The user uploads a PDF document (a contract, a financial report, a technical spec — something with real tables and structure). The system ingests it in two parallel ways:

**Path A — Standard flat RAG:**

- Split the document into fixed-size token chunks (512 tokens, 50-token overlap)
- Embed each chunk using `text-embedding-3-small`
- Store in a local vector store (ChromaDB)
- At query time: embed the question, retrieve top-5 chunks, pass to GPT-4o, return answer

**Path B — Structure-Aware ingest:**

- Parse the PDF using `pdfplumber` to extract text blocks, tables (as structured data), and section headers separately
- Store tables as JSON objects with row/column structure, not as flattened text
- Store text sections with metadata: section title, page number, and any table/figure references they contain
- Resolve cross-references: if a text chunk says "see Table 3", tag it with a link to the Table 3 object
- Embed each element separately. Tables get embedded as a natural-language summary of their content generated by GPT-4o-mini, plus the raw JSON is stored alongside
- At query time: classify the question (is it asking about a table? a section? a specific clause?), retrieve accordingly, reconstruct the full structural context before passing to GPT-4o

**The demo:**
Ask the same 5 questions to both systems. Questions should be designed to expose flat RAG's weaknesses:

- "What was the total revenue in Q3?" (requires reading a table correctly)
- "Under what conditions can the contract be terminated early?" (requires reading a specific clause with correct context)
- "What is the maximum liability cap, and how was it calculated?" (requires combining a table cell with a clause explanation)
- "Which sections reference the payment schedule?" (requires cross-reference resolution)
- "Summarize the key financial obligations in this agreement" (requires understanding structure across the document)

Display the two answers side by side. Below each answer, show the source chunks that were retrieved, so the user can see that flat RAG retrieved a fragment while structure-aware ingest retrieved the complete table or clause.

### Module 2: 5-Dimension Runtime Confidence Scorer (The Confidence Pillar)

**What it does:**

For every answer produced by either RAG pipeline, the system runs a confidence scoring step that evaluates the answer across 5 dimensions. Each dimension gets a score from 0 to 1 and a short explanation. The scores combine into an overall confidence signal.

**The 5 Dimensions:**

1. **Source Grounding** — Does every factual claim in the answer trace back to a retrieved chunk? Use an LLM judge that is given the answer and the retrieved chunks. It should identify claims that have no source chunk supporting them. Score = (grounded claims) / (total claims).

2. **Retrieval Completeness** — Were enough relevant chunks retrieved to fully answer the question? Use an LLM judge that is given the question and the retrieved chunks. It should evaluate whether there are obvious gaps — parts of the question that none of the retrieved chunks address. Score = 0 to 1 based on judge output.

3. **Factual Consistency** — Does the answer contradict any of the retrieved source chunks? The judge is given the answer and the chunks and asked to identify any contradictions. Score = 1 if no contradictions, decreasing with each one found.

4. **Structural Completeness** — For questions that require reading a table (detected by classifying the question), was a structured table chunk retrieved, or only flat text? Score = 1 if a structured table was retrieved for a table-dependent question, 0.3 if only flat text was retrieved for the same question.

5. **Confidence Calibration** — Does the model's phrasing match its actual reliability? If the model says "the total is exactly $4.2M" but the source grounding score is below 0.5, this dimension flags an overconfident response. If the model appropriately hedges ("based on the retrieved sections, it appears..."), this scores well. Use a simple rule-based check on hedge language vs. grounding score delta.

**The output:**
Each answer card shows a radar chart of the 5 scores, an overall confidence percentage, and a plain-English summary of the weakest dimension with a recommendation ("Answer may be incomplete — retrieval did not return a structured table for this financial question. Recommend re-querying with table-aware retrieval.").

### What This Demonstrates Together

When you run both modules together, the demo tells a clear story:

- Structure-aware ingest produces better answers on structured questions (Module 1)
- The confidence scorer can _detect_ when flat RAG has failed, even without ground truth (Module 2)
- Every answer is traceable to its source — you can see exactly which chunk was retrieved and why
- The system improves in a measurable, visible way — this is not vibe-based quality improvement

This demonstrates a fundamental principle: reliability is an infrastructure problem, solved at ingest and instrumented at runtime, not fixed by better prompts.

## Technical Architecture

### Backend — Python (FastAPI)

**Why Python and not Go for this project:**
The demo requires `pdfplumber` for structured PDF parsing, `chromadb` for local vector storage, and direct OpenAI SDK calls for embeddings and chat completions. These are all mature Python libraries with no equivalent in Go. Go is the right choice for a service mesh or API gateway; Python is the right choice for an ML/data pipeline demo.

- Framework: FastAPI with Uvicorn
- PDF parsing: `pdfplumber` (preserves table structure as lists of rows)
- Vector store: `chromadb` (local, in-memory for demo)
- Embeddings: OpenAI `text-embedding-3-small` via `openai` SDK
- LLM: OpenAI `gpt-4o` for main answers, `gpt-4o-mini` for judge calls (cheaper)
- API key loaded from environment variable `OPENAI_API_KEY`

**Endpoints:**

| Method | Path             | Description                                                            |
| ------ | ---------------- | ---------------------------------------------------------------------- |
| `POST` | `/api/ingest`    | Upload PDF, run both flat and structure-aware ingest pipelines         |
| `POST` | `/api/query`     | Ask a question, get answers from both pipelines with confidence scores |
| `GET`  | `/api/documents` | List ingested documents                                                |
| `GET`  | `/health`        | Health check                                                           |

### Frontend — Minimal HTML + Vanilla JS

- Single HTML file served by FastAPI's `StaticFiles`
- Three sections: Upload, Query, Results
- Results section shows two columns (Flat RAG vs. Structure-Aware) side by side
- Each answer card includes: the answer text, retrieved source chunks (collapsed by default), and the 5-dimension confidence scorecard with a mini bar chart per dimension
- Tailwind CSS via CDN
- No build toolchain

### Step 3 — LLM Client

Create `llm/client.py` with three functions:

**`embed(text: str) -> List[float]`**

- Call OpenAI Embeddings API with `text-embedding-3-small`
- Return the embedding vector
- Cache embeddings in a simple in-memory dict keyed by SHA256(text) to avoid re-embedding the same text

**`complete(system_prompt: str, user_message: str, model: str = "gpt-4o") -> str`**

- Call OpenAI Chat Completions API
- Set `temperature=0` for deterministic outputs
- Return the assistant message content string

**`judge(instruction: str, context: str, model: str = "gpt-4o-mini") -> dict`**

- Specialized variant of `complete` for structured judge calls
- Appends "Respond with valid JSON only." to the system prompt
- Parses and returns the response as a Python dict
- On JSON parse failure, retry once with a stricter prompt; on second failure, return `{"error": "parse_failed"}`

### Step 4 — Flat RAG Ingest Pipeline

Create `ingest/flat_rag.py`.

**Function: `ingest_flat(doc_id: str, pdf_path: str, chroma_client) -> List[FlatChunk]`**

Steps:

1. Open the PDF with `pdfplumber`
2. Extract all text page by page using `page.extract_text()`
3. Concatenate all pages into one text string, tracking page boundaries
4. Chunk the text: split on sentence boundaries (split on `. ` and `\n\n`), then group sentences into chunks of approximately 400 words with a 50-word overlap between consecutive chunks
5. For each chunk, create a `FlatChunk` with a UUID, doc_id, text, page number (estimate from character offset vs. page boundary map), and chunk index
6. Embed each chunk's text using `llm.client.embed()`
7. Store in ChromaDB collection named `f"flat_{doc_id}"` with metadata `{chunk_id, page_number, chunk_index}`
8. Return the list of `FlatChunk` objects

**Function: `retrieve_flat(question: str, doc_id: str, chroma_client, top_k: int = 5) -> List[FlatChunk]`**

1. Embed the question
2. Query the ChromaDB collection `f"flat_{doc_id}"` for top_k nearest chunks
3. Return the chunks in ranked order

### Step 5 — Structure-Aware Ingest Pipeline

Create `ingest/structured_rag.py`. This is the most important file in the project.

**Function: `ingest_structured(doc_id: str, pdf_path: str, chroma_client) -> tuple[List[TableChunk], List[TextSectionChunk]]`**

Steps:

**5a — Extract tables:**

1. Open the PDF with `pdfplumber`
2. For each page, call `page.extract_tables()` — this returns tables as `List[List[str]]` (a list of rows, each row a list of cell strings)
3. For each extracted table:
   - Find the nearest text above the table on the same page to use as the section title (scan upward for the last non-empty line of text)
   - Clean the table: strip whitespace from cells, fill empty cells with `""`, handle merged cells by forward-filling the value
   - Generate a natural language summary by calling `llm.client.complete()` with a prompt like: "You are given a table extracted from a document. Summarize what this table shows in 2-3 sentences, describing its structure, what each column represents, and the range or key values in each column. Table: {json.dumps(rows)}"
   - Create a `TableChunk` with a UUID, the rows, the summary, and the summary as `embedding_text`
   - Embed the summary
   - Store in ChromaDB collection `f"tables_{doc_id}"` with metadata `{chunk_id, page_number, section_title}`

**5b — Extract text sections:**

1. For each page, call `page.extract_text()` to get raw text
2. Split into sections by detecting section headers: lines that are all-caps, or lines followed by two newlines, or lines that match common heading patterns (`^\d+\.\s`, `^[A-Z][A-Z\s]+$`)
3. For each section:
   - Store the section title (the detected header line) and the body text
   - Detect table cross-references: scan the body text for patterns like "Table \d+", "see table", "as shown in the table above" — record these as `referenced_table_ids` by matching against the table chunks extracted from the same or adjacent pages
   - Create a `TextSectionChunk`
   - Embed the text
   - Store in ChromaDB collection `f"sections_{doc_id}"` with metadata `{chunk_id, page_number, section_title}`

Return the lists of all `TableChunk` and `TextSectionChunk` objects.

### Step 6 — Structure-Aware Retrieval

Create `retrieval/structured_retriever.py`.

**Function: `classify_question(question: str) -> str`**

Use a simple LLM call to classify the question into one of three types:

- `"table_query"` — the question is asking for a specific number, comparison, or data point likely found in a table (e.g., "what was the revenue", "how many", "what is the maximum")
- `"clause_query"` — the question is asking about conditions, terms, policies, or requirements (e.g., "under what conditions", "what happens if", "what are the requirements for")
- `"summary_query"` — the question asks for a summary, overview, or synthesis across the document

Return the classification string.

**Function: `retrieve_structured(question: str, doc_id: str, chroma_client, top_k: int = 5) -> List[Union[TableChunk, TextSectionChunk]]`**

1. Classify the question
2. If `"table_query"`:
   - Query `f"tables_{doc_id}"` for top-3 table chunks
   - Query `f"sections_{doc_id}"` for top-2 section chunks that might provide context
   - For each retrieved section, also include any `TableChunk` objects linked by `referenced_table_ids`
   - Return the combined, deduplicated list ranked by relevance
3. If `"clause_query"`:
   - Query `f"sections_{doc_id}"` for top-5 section chunks
   - For each retrieved section, include any linked table chunks via `referenced_table_ids`
   - Return the combined list
4. If `"summary_query"`:
   - Query both collections for top-3 each
   - Return the combined top-5 by score

### Step 7 — Confidence Scorer

Create each file in `confidence/`.

**`confidence/source_grounding.py`**

Function: `score_source_grounding(answer: str, retrieved_chunks: List) -> DimensionScore`

Prompt for the judge model:

```
You are evaluating whether an AI answer is fully supported by the retrieved source documents.

Retrieved sources:
{formatted_chunks}

AI Answer:
{answer}

Identify each factual claim in the answer. For each claim, determine if it is directly supported by one of the retrieved sources.

Respond with valid JSON:
{
  "total_claims": <int>,
  "grounded_claims": <int>,
  "unsupported_claims": [<list of claim strings that have no source>],
  "score": <float 0.0 to 1.0>
}
```

Compute `score = grounded_claims / total_claims`. If total_claims is 0, score = 1.0.

Return a `DimensionScore` with the score and an explanation summarizing any unsupported claims.

**`confidence/retrieval_completeness.py`**

Function: `score_retrieval_completeness(question: str, retrieved_chunks: List) -> DimensionScore`

Prompt:

```
You are evaluating whether the retrieved documents contain enough information to fully answer a question.

Question: {question}

Retrieved documents:
{formatted_chunks}

Assess whether the retrieved documents address all parts of the question. Are there aspects of the question that none of the documents speak to?

Respond with valid JSON:
{
  "fully_addressed": <bool>,
  "missing_aspects": [<list of aspects of the question not covered>],
  "score": <float 0.0 to 1.0>
}
```

**`confidence/factual_consistency.py`**

Function: `score_factual_consistency(answer: str, retrieved_chunks: List) -> DimensionScore`

Prompt:

```
You are checking whether an AI answer contradicts any of the source documents it was based on.

Source documents:
{formatted_chunks}

AI Answer:
{answer}

Identify any statements in the answer that directly contradict information in the source documents. A contradiction means the answer states something that the source document says the opposite of.

Respond with valid JSON:
{
  "contradictions_found": <int>,
  "contradictions": [<list of contradiction descriptions>],
  "score": <float: 1.0 if 0 contradictions, 0.7 for 1, 0.4 for 2, 0.0 for 3+>
}
```

**`confidence/structural_completeness.py`**

Function: `score_structural_completeness(question: str, retrieved_chunks: List, question_type: str) -> DimensionScore`

Rule-based (no LLM call):

- If `question_type != "table_query"`, return score = 1.0 with explanation "Not applicable for this question type"
- If `question_type == "table_query"`: check if any chunk in `retrieved_chunks` is a `TableChunk`
  - If yes: score = 1.0, explanation = "A structured table was retrieved for this data question"
  - If no: score = 0.3, explanation = "This question asks for tabular data but only flat text was retrieved — the answer may be incomplete or incorrect"

**`confidence/confidence_calibration.py`**

Function: `score_confidence_calibration(answer: str, source_grounding_score: float) -> DimensionScore`

Rule-based:

- Scan the answer for hedge phrases: "appears to", "based on the available", "approximately", "may be", "it seems", "I cannot confirm", "the document suggests"
- Scan for overconfidence phrases: "exactly", "the answer is", "the total is", "definitively", "clearly states", "is exactly"
- If `source_grounding_score < 0.6` and the answer contains overconfidence phrases: score = 0.2, explanation = "Answer uses definitive language but source grounding is weak"
- If `source_grounding_score < 0.6` and the answer uses hedge phrases: score = 0.9, explanation = "Answer appropriately hedges given uncertain retrieval"
- If `source_grounding_score >= 0.8` and the answer uses hedge phrases: score = 0.7, explanation = "Answer is overly cautious given strong source grounding"
- If `source_grounding_score >= 0.8` and no problematic phrases: score = 1.0, explanation = "Answer confidence aligns with retrieval quality"

**`confidence/scorer.py`**

Function: `score(question: str, answer: str, retrieved_chunks: List, question_type: str) -> ConfidenceReport`

Calls all 5 dimension scorers, computes `overall_score` as the weighted average (source grounding: 0.30, retrieval completeness: 0.25, factual consistency: 0.25, structural completeness: 0.10, confidence calibration: 0.10), finds the weakest dimension, and generates a plain-English recommendation for it.

### Step 8 — API Endpoints

Create route handlers in `main.py` (or in separate router files imported into `main.py`).

**`POST /api/ingest`**

- Accept a multipart file upload (`UploadFile`)
- Save the file temporarily to `/tmp/{uuid}_{filename}`
- Generate a `doc_id` (UUID)
- Run `ingest_flat(doc_id, path, chroma_client)` and `ingest_structured(doc_id, path, chroma_client)` — these can run sequentially
- Store the `IngestedDocument` in a simple in-memory dict keyed by `doc_id`
- Clean up the temp file
- Return `{"doc_id": doc_id, "filename": filename, "flat_chunks": len(...), "table_chunks": len(...), "section_chunks": len(...)}`

**`POST /api/query`**

- Accept JSON body: `{"doc_id": "...", "question": "..."}`
- Validate that the document exists in the in-memory store
- Run flat retrieval and structured retrieval in parallel (use `asyncio.gather` with `run_in_executor` since the ChromaDB and LLM calls are blocking)
- For each pipeline:
  - Generate an answer by calling `llm.client.complete()` with a system prompt like: "You are answering a question based strictly on the provided source documents. If the information needed to answer is not in the sources, say so explicitly. Do not invent information." and a user message that includes the formatted retrieved chunks and the question
  - Run the confidence scorer
  - Build a `PipelineAnswer`
- Return a `QueryResponse` serialized as JSON

**`GET /api/documents`**

- Return a list of all ingested documents: `[{"doc_id": ..., "filename": ..., "ingested_at": ...}]`

- The frontend should be usable without any explanation — a non-technical person should be able to upload a PDF, ask a question, and immediately understand the comparison
- The confidence scores should be _calibrated_ — they should genuinely reflect the quality difference between the two pipelines. Test this by verifying that structure-aware retrieval scores consistently higher than flat RAG on table-dependent questions
- The LLM prompts should always request JSON and the client should always validate the JSON before using it. Never trust raw LLM output without parsing it.
- API error responses should always be JSON: `{"error": "message", "detail": "..."}`
